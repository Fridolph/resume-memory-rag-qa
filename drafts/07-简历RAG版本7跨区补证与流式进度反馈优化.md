# 第 07 章：简历 RAG 版本 7 的跨区补证与流式进度反馈优化

## 背景

版本 6 已经做出了很明显的进步：

- 引入了 `primary / support` 的证据分层
- `LC 安全大屏` 这类无关项目被更稳定地压制
- 最终进入 Prompt 的证据明显更干净

但这轮真实运行结果也暴露了几个新问题。

---

## 这轮 v6 反馈的关键结论

### 1. `skills / AI Agent` 这类高相关片段被过滤得太狠

这一轮最值得注意的现象是：

- 某些 `skills` 片段虽然：
  - `raw score` 很高
  - `topicHit = true`
  - 语义上也和问题高度相关
- 但只因为它不属于经验类问题下的优先 section，
  最终就直接被丢掉

这说明版本 6 在“主证据优先”上是成功的，  
但同时也出现了新的副作用：

> **它对非优先 section 的压制过强，导致一些“高相关补充证据”也一起被误杀了。**

换句话说：

- `skills` 不应该抢主位
- 但它也不应该在“高度相关”时完全失声

---

### 2. `my-resume` 已进入 final，但回答里仍可能没被点名

版本 6 的另一个观察是：

- `my-resume` 虽然已经进入 `finalMatches`
- 但模型最终回答时，仍然可能没有把它单独讲出来

这说明仅仅把片段放进 Prompt 还不够，  
还需要进一步引导模型理解：

> **辅助证据里如果存在“和主证据不同的独立项目 / 工作台 / 能力点”，就应该主动点名，而不是默认忽略。**

这本质上已经不只是“召回问题”，  
而是：

- **证据组织方式**
- **Prompt 使用方式**

两者共同作用的问题。

---

### 3. `finalMatches` 过少，虽然更精，但信息面变窄

版本 6 的设计倾向是：

- 少一点
- 但尽量更纯

这在很多情况下是对的，  
但这轮结果里也出现了新的问题：

- `topK = 8`
- 最终真正进入 Prompt 的却只有 4 条左右

这会带来一个新的风险：

> **上下文虽然更干净了，但信息覆盖面也变窄了。**

尤其在简历问答里，  
我们经常希望模型既能回答“最主要的代表经历”，  
又能补充：

- 另一个相关项目
- 一个高相关技能点
- 一个辅助的工作或平台线索

所以版本 7 的目标不是简单把结果“再放宽”，  
而是：

> **在不放回明显噪声的前提下，让 final context 保持一个更合理的最小信息量。**

---

### 4. 运行过程没有过渡反馈，CLI 体验很差

这个问题非常真实，也很重要。

当前运行：

```bash
node ./src/ask-resume-6-evidence-ask.mjs
```

在连接完成后，会看到：

- Session 信息
- Cache 信息

然后就进入长时间等待，  
直到最终结果一次性全部输出。

这在 CLI 里会带来两个非常差的体验：

1. **不知道程序是不是卡住了**
2. **即使没卡，也感知不到“当前进展到哪一步了”**

这和后续我们要做的：

- SSE
- Web 端流式回答
- 更友好的 Agent 交互

其实是同一类问题。

因此版本 7 会把“过程感知”也纳入优化目标。

---

## 版本 7 的核心优化目标

基于这轮反馈，版本 7 聚焦 4 个点：

### 1. 引入“跨区补证（cross-section support）”

版本 7 不再把证据简单分成：

- `primary`
- `support`
- `dropped`

而是增加一个新层：

- `cross_support`

它的定位是：

> **虽然这条证据不在当前问题的优先 section，但它与问题主题高度相关，且得分足够高，所以允许它作为补充证据保留下来。**

典型例子就是：

- `skills / AI Agent 开发`

它不应该替代：

- `projects`
- `work_experience`

但可以作为：

- “补充佐证”
- “能力侧证明”

存在于最终上下文中。

---

### 2. Prompt 明确要求：辅助证据中的独立项目不要忽略

版本 7 不只是在检索层修补，  
还在 Prompt 层进一步约束模型：

- 先讲 `primary`
- 再看 `support`
- 如果 `support` / `cross_support` 中存在与主证据不同的独立项目或工作台，
  必须单独补充点名

这就是为了减少类似：

- `my-resume` 明明进了 final
- 但最后回答没提到

这样的情况。

---

### 3. 增加“最小 final 数量”与“兜底补充”

版本 7 不再默认接受：

- 只剩 4 条也算完成

而是增加一层控制：

- `minFinalCount`
- `reserveMatches`

当：

- `primary + support + cross_support`

还不够时，  
允许再补入一小部分：

- 分数还不错
- 且不是明显噪声

的 `reserve evidence`，  
以保证最终上下文不要过瘦。

这样做的目标不是让 Prompt 重新变脏，  
而是：

> **在“纯度”和“信息覆盖”之间重新找平衡。**

---

### 4. 加入阶段日志与流式回答

这轮版本 7 不再只关心“最终结果对不对”，  
也开始关心：

> **运行过程中，用户能否感知到系统正在做什么。**

因此新脚本会在 CLI 中提供两种反馈：

#### （1）阶段日志

例如：

- `1/7 策略识别完成`
- `2/7 已读取集合 schema`
- `3/7 已完成向量召回`
- `4/7 已完成重排`
- `5/7 证据分层完成`
- `6/7 正在构建 Prompt messages`
- `7/7 已开始流式生成回答`

这样即使模型还没答完，  
你也能知道它当前卡在：

- 检索
- 重排
- 还是生成

#### （2）流式输出回答

版本 7 默认用 `model.stream()` 来输出回答，  
这样在 CLI 中可以更像后续 Web / SSE 交互：

- 回答不是最后一次性打印
- 而是边生成边显示

这一步虽然还是 CLI 版 demo，  
但它已经在交互体验上往“真实产品形态”更靠近了一步。

---

## 版本 7 的整体处理流程

```text
raw recall
  ↓
rerank
  ↓
denoise
  ↓
primary evidence
support evidence
cross-section support
reserve evidence
  ↓
final context
  ↓
prompt messages
  ↓
stream answer
```

和版本 6 相比，最重要的新增有两个：

1. **`cross-section support`**
2. **`stream answer + progress log`**

---

## 这轮的核心设计取舍

版本 7 并不是否定版本 6，  
而是在版本 6 的基础上修正两个偏差：

### 偏差一：太强调“主证据优先”，导致高相关补充证据也被压掉

修正方式：

- 增加 `cross_support`

### 偏差二：太强调“最终结果”，忽略运行过程的可感知性

修正方式：

- 增加 progress logs
- 增加流式回答

因此版本 7 的一句话总结可以写成：

> **在延续版本 6“主证据优先”的前提下，版本 7 进一步解决了“高相关补充证据被误杀”和“CLI 等待过程缺少反馈”这两个实际体验问题。**

---

## 当前一次实跑后的进一步观察

版本 7 首次实跑后，又得到了一组很有价值的观察：

### 1. 阶段日志已经明显改善了“等待感”

这次运行时，CLI 会连续输出：

- `1/7 策略识别完成`
- `2/7 已读取集合 schema`
- `3/7 已完成向量召回`
- `4/7 已完成重排`
- `5/7 证据分层完成`
- `6/7 正在构建 Prompt messages`
- `7/7 正在生成 / 流式生成回答`

这说明版本 7 至少已经解决了一个非常实际的问题：

> **用户不再需要盲等，也能感知“现在卡在检索、重排还是生成”。**

### 2. `my-resume` 已经能更稳定地进入回答正文

这次实跑中，`my-resume` 不仅进入了 `support`，  
而且回答里已经能被明确点名为：

- `AI 分析工作台`

这说明版本 7 在 Prompt 层加的这条约束是有效的：

> **如果 support evidence 中包含和主证据不同的独立项目 / 工作台，就不要忽略。**

### 3. 但第一版阈值下，`skills` 还没有真正进入 `cross_support`

虽然版本 7 已经引入了 `cross_support` 机制，  
但首次实跑里：

- 高相关 `skills / AI Agent 开发`

- `reserve`

而不是：

- `cross_support`

这说明第一版阈值还偏保守。

### 4. `reserve` 中仍可能混入边缘项目

首次实跑里还看到：

- `EDR`

这类并不是当前问题核心主题的片段，  
仍可能通过 `reserve` 进入最终上下文。

这意味着：

> **v7 的方向是对的，但阈值还需要再收一下。**

## 基于实跑结果的第二次小调优

因此版本 7 在首轮实跑后，继续做了两处很小但很关键的参数修正：

### 1. 降低 `cross_support` 的 `rerank` 门槛

把经验类问题下：

- `crossSectionSupportMinRerankScore`

从：

- `0.68`

调到：

- `0.62`

目的很明确：

> **让“raw 很高、topicHit 很强、但因为 section 不优先而被压住”的 skills 片段，真正进入 `cross_support`。**

### 2. 提高 `reserve` 的最低分

把经验类问题下：

- `reserveMinRerankScore`

从：

- `0.58`

调到：

- `0.60`

目的也很明确：

> **把像 `EDR` 这种边缘片段继续往外压，尽量让 reserve 保留“还有一定相关性”的补充证据。**

---

## 从 v6 的 `rag-pipeline` 到 v7：继续收缩为“流程编排器”

这一轮还有一个很值得记录的工程点，不是“召回效果”本身，  
而是 `rag-pipeline` 这个模块在架构上的继续收缩。

### v4 / v5 早期的情况

早期版本里，`rag-pipeline` 往往同时承担很多职责：

- 策略识别
- 关键词扩展
- section boost 计算
- rerank
- context 格式化
- Prompt 构造
- 最终模型调用

这会带来一个问题：

> **一个文件里同时塞了“检索策略逻辑”和“生成编排逻辑”，阅读和维护成本会越来越高。**

### 到了 v6 / v7 的变化

到了版本 6、版本 7，`rag-pipeline` 已经越来越像一个真正的“流程编排器”：

- `context-builder` 负责准备证据
- `prompt-builder` 负责准备 Prompt / messages
- `rag-pipeline` 只负责把这几个环节串起来，并调用模型

如果用一句更工程化的话说，就是：

> **pipeline 不再负责“怎么算”，而只负责“按什么顺序调用”。**

这其实就是单一职责原则在这套 demo 里的一个很直观的落地过程。

---

## 这轮顺手修掉的两个 pipeline 工程问题

### 1. 补上 LLM 调用的错误处理

你这次提到的这个点非常关键。

像下面这种代码：

```js
const response = await model.invoke(messages)
```

如果出现：

- 超时
- 限流
- 网关错误
- 返回空内容

那么错误会直接从底层冒上来，  
但在学习阶段你往往更希望看到：

- 是检索失败了？
- 是 Prompt 构造失败了？
- 还是 LLM 调用失败了？

#### 这次怎么改

版本 7 里把 `runRAGv7()` 这层补上了更明确的错误边界：

- `buildRAGContextV7()` 失败 → `RAG 上下文构建失败`
- `buildMessagesV7()` 失败 → `Prompt messages 构建失败`
- `model.stream()` 失败 → `LLM 流式调用失败`
- `model.invoke()` 失败 → `LLM 调用失败`
- 返回空内容 → `LLM 返回了空内容`

这样后面排查时会更快定位问题到底出在哪一层。

### 2. 把 `candidateTopK` 的默认值统一留在 `context-builder`

这个点也很重要，因为它属于“隐藏歧义”。

如果：

- `runRAGv7()` 自己有一套默认值
- `buildRAGContextV7()` 里又有另一套默认值

那后面哪怕只有一边改了，  
都会出现：

> **调用方以为默认是 A，实际执行时却落成了 B。**

#### 这次怎么改

这次把 `runRAGv7()` 里的：

- `candidateTopK = Math.max(topK * 2, 10)`

去掉了，改成直接透传。

现在统一由：

- `buildRAGContextV7()`

内部处理默认值。

这样默认值逻辑只有一个来源，  
后面再调参时不会出现两边不同步的问题。

## 关于 `noCache` 日志这件事

你提到：

> `noCache` 虽然透传了，但 pipeline 本身不感知，最好在调用层打印一次。

这个建议是对的。

这件事最终最适合放在：

- CLI 入口脚本

而不是放在 pipeline 里。

原因是：

- `pipeline` 更偏流程编排
- `ask` 脚本才是用户真正能看到输出的地方

因此版本 7 的入口脚本已经明确打印：

- `Config cache: enabled`
- 或 `Config cache: disabled (--no-cache)`

这样既满足可见性，  
又不会把“CLI 表现层日志”塞回 pipeline。
