# 第 07 章：简历 RAG 版本 7 的跨区补证与流式进度反馈优化

## 背景

版本 6 已经做出了很明显的进步：

- 引入了 `primary / support` 的证据分层
- `LC 安全大屏` 这类无关项目被更稳定地压制
- 最终进入 Prompt 的证据明显更干净

但这轮真实运行结果也暴露了几个新问题。

---

## 这轮 v6 反馈的关键结论

### 1. `skills / AI Agent` 这类高相关片段被过滤得太狠

这一轮最值得注意的现象是：

- 某些 `skills` 片段虽然：
  - `raw score` 很高
  - `topicHit = true`
  - 语义上也和问题高度相关
- 但只因为它不属于经验类问题下的优先 section，
  最终就直接被丢掉

这说明版本 6 在“主证据优先”上是成功的，  
但同时也出现了新的副作用：

> **它对非优先 section 的压制过强，导致一些“高相关补充证据”也一起被误杀了。**

换句话说：

- `skills` 不应该抢主位
- 但它也不应该在“高度相关”时完全失声

---

### 2. `my-resume` 已进入 final，但回答里仍可能没被点名

版本 6 的另一个观察是：

- `my-resume` 虽然已经进入 `finalMatches`
- 但模型最终回答时，仍然可能没有把它单独讲出来

这说明仅仅把片段放进 Prompt 还不够，  
还需要进一步引导模型理解：

> **辅助证据里如果存在“和主证据不同的独立项目 / 工作台 / 能力点”，就应该主动点名，而不是默认忽略。**

这本质上已经不只是“召回问题”，  
而是：

- **证据组织方式**
- **Prompt 使用方式**

两者共同作用的问题。

---

### 3. `finalMatches` 过少，虽然更精，但信息面变窄

版本 6 的设计倾向是：

- 少一点
- 但尽量更纯

这在很多情况下是对的，  
但这轮结果里也出现了新的问题：

- `topK = 8`
- 最终真正进入 Prompt 的却只有 4 条左右

这会带来一个新的风险：

> **上下文虽然更干净了，但信息覆盖面也变窄了。**

尤其在简历问答里，  
我们经常希望模型既能回答“最主要的代表经历”，  
又能补充：

- 另一个相关项目
- 一个高相关技能点
- 一个辅助的工作或平台线索

所以版本 7 的目标不是简单把结果“再放宽”，  
而是：

> **在不放回明显噪声的前提下，让 final context 保持一个更合理的最小信息量。**

---

### 4. 运行过程没有过渡反馈，CLI 体验很差

这个问题非常真实，也很重要。

当前运行：

```bash
node ./src/ask-resume-6-evidence-ask.mjs
```

在连接完成后，会看到：

- Session 信息
- Cache 信息

然后就进入长时间等待，  
直到最终结果一次性全部输出。

这在 CLI 里会带来两个非常差的体验：

1. **不知道程序是不是卡住了**
2. **即使没卡，也感知不到“当前进展到哪一步了”**

这和后续我们要做的：

- SSE
- Web 端流式回答
- 更友好的 Agent 交互

其实是同一类问题。

因此版本 7 会把“过程感知”也纳入优化目标。

---

## 版本 7 的核心优化目标

基于这轮反馈，版本 7 聚焦 4 个点：

### 1. 引入“跨区补证（cross-section support）”

版本 7 不再把证据简单分成：

- `primary`
- `support`
- `dropped`

而是增加一个新层：

- `cross_support`

它的定位是：

> **虽然这条证据不在当前问题的优先 section，但它与问题主题高度相关，且得分足够高，所以允许它作为补充证据保留下来。**

典型例子就是：

- `skills / AI Agent 开发`

它不应该替代：

- `projects`
- `work_experience`

但可以作为：

- “补充佐证”
- “能力侧证明”

存在于最终上下文中。

---

### 2. Prompt 明确要求：辅助证据中的独立项目不要忽略

版本 7 不只是在检索层修补，  
还在 Prompt 层进一步约束模型：

- 先讲 `primary`
- 再看 `support`
- 如果 `support` / `cross_support` 中存在与主证据不同的独立项目或工作台，
  必须单独补充点名

这就是为了减少类似：

- `my-resume` 明明进了 final
- 但最后回答没提到

这样的情况。

---

### 3. 增加“最小 final 数量”与“兜底补充”

版本 7 不再默认接受：

- 只剩 4 条也算完成

而是增加一层控制：

- `minFinalCount`
- `reserveMatches`

当：

- `primary + support + cross_support`

还不够时，  
允许再补入一小部分：

- 分数还不错
- 且不是明显噪声

的 `reserve evidence`，  
以保证最终上下文不要过瘦。

这样做的目标不是让 Prompt 重新变脏，  
而是：

> **在“纯度”和“信息覆盖”之间重新找平衡。**

---

### 4. 加入阶段日志与流式回答

这轮版本 7 不再只关心“最终结果对不对”，  
也开始关心：

> **运行过程中，用户能否感知到系统正在做什么。**

因此新脚本会在 CLI 中提供两种反馈：

#### （1）阶段日志

例如：

- `1/7 策略识别完成`
- `2/7 已读取集合 schema`
- `3/7 已完成向量召回`
- `4/7 已完成重排`
- `5/7 证据分层完成`
- `6/7 正在构建 Prompt messages`
- `7/7 已开始流式生成回答`

这样即使模型还没答完，  
你也能知道它当前卡在：

- 检索
- 重排
- 还是生成

#### （2）流式输出回答

版本 7 默认用 `model.stream()` 来输出回答，  
这样在 CLI 中可以更像后续 Web / SSE 交互：

- 回答不是最后一次性打印
- 而是边生成边显示

这一步虽然还是 CLI 版 demo，  
但它已经在交互体验上往“真实产品形态”更靠近了一步。

---

## 版本 7 的整体处理流程

```text
raw recall
  ↓
rerank
  ↓
denoise
  ↓
primary evidence
support evidence
cross-section support
reserve evidence
  ↓
final context
  ↓
prompt messages
  ↓
stream answer
```

和版本 6 相比，最重要的新增有两个：

1. **`cross-section support`**
2. **`stream answer + progress log`**

---

## 这轮的核心设计取舍

版本 7 并不是否定版本 6，  
而是在版本 6 的基础上修正两个偏差：

### 偏差一：太强调“主证据优先”，导致高相关补充证据也被压掉

修正方式：

- 增加 `cross_support`

### 偏差二：太强调“最终结果”，忽略运行过程的可感知性

修正方式：

- 增加 progress logs
- 增加流式回答

因此版本 7 的一句话总结可以写成：

> **在延续版本 6“主证据优先”的前提下，版本 7 进一步解决了“高相关补充证据被误杀”和“CLI 等待过程缺少反馈”这两个实际体验问题。**

## 从 v6 的 `rag-pipeline` 到 v7：继续收缩为“流程编排器”

这一轮还有一个很值得记录的工程点，不是“召回效果”本身，  
而是 `rag-pipeline` 这个模块在架构上的继续收缩。

### v4 / v5 早期的情况

早期版本里，`rag-pipeline` 往往同时承担很多职责：

- 策略识别
- 关键词扩展
- section boost 计算
- rerank
- context 格式化
- Prompt 构造
- 最终模型调用

这会带来一个问题：

> **一个文件里同时塞了“检索策略逻辑”和“生成编排逻辑”，阅读和维护成本会越来越高。**

### 到了 v6 / v7 的变化

到了版本 6、版本 7，`rag-pipeline` 已经越来越像一个真正的“流程编排器”：

- `context-builder` 负责准备证据
- `prompt-builder` 负责准备 Prompt / messages
- `rag-pipeline` 只负责把这几个环节串起来，并调用模型

如果用一句更工程化的话说，就是：

> **pipeline 不再负责“怎么算”，而只负责“按什么顺序调用”。**

这其实就是单一职责原则在这套 demo 里的一个很直观的落地过程。

## 这轮顺手修掉的两个 pipeline 工程问题

### 1. 补上 LLM 调用的错误处理

你这次提到的这个点非常关键。

像下面这种代码：

```js
const response = await model.invoke(messages)
```

如果出现：

- 超时
- 限流
- 网关错误
- 返回空内容

那么错误会直接从底层冒上来，  
但在学习阶段你往往更希望看到：

- 是检索失败了？
- 是 Prompt 构造失败了？
- 还是 LLM 调用失败了？

#### 这次怎么改

版本 7 里把 `runRAGv7()` 这层补上了更明确的错误边界：

- `buildRAGContextV7()` 失败 → `RAG 上下文构建失败`
- `buildMessagesV7()` 失败 → `Prompt messages 构建失败`
- `model.stream()` 失败 → `LLM 流式调用失败`
- `model.invoke()` 失败 → `LLM 调用失败`
- 返回空内容 → `LLM 返回了空内容`

这样后面排查时会更快定位问题到底出在哪一层。

### 2. 把 `candidateTopK` 的默认值统一留在 `context-builder`

这个点也很重要，因为它属于“隐藏歧义”。

如果：

- `runRAGv7()` 自己有一套默认值
- `buildRAGContextV7()` 里又有另一套默认值

那后面哪怕只有一边改了，  
都会出现：

> **调用方以为默认是 A，实际执行时却落成了 B。**

#### 这次怎么改

这次把 `runRAGv7()` 里的：

- `candidateTopK = Math.max(topK * 2, 10)`

去掉了，改成直接透传。

现在统一由：

- `buildRAGContextV7()`

内部处理默认值。

这样默认值逻辑只有一个来源，  
后面再调参时不会出现两边不同步的问题。

## 这轮继续修的 v7 技术债：去掉跨版本依赖

这次还收到了一个非常关键的反馈：

```js
import { denoiseMatches } from '../rag4/rag-pipeline-v4.mjs'
import { detectQuestionStrategy } from '../rag5/context-builder.mjs'
```

这种写法虽然“能跑”，  
但本质上已经形成了明显的跨版本耦合：

- v7 的核心逻辑还在依赖 v4 / v5
- 你读 v7 时，还得跳回老版本看关键实现
- 如果老版本函数签名改了，v7 可能会被悄悄带坏

这类问题在学习阶段尤其危险，因为它会让人误以为：

> **“版本号在升级”，但核心逻辑其实并没有在当前版本内部闭合。**

### 这次怎么修

这次在 `rag7/context-builder-v7.mjs` 里做了两件事：

#### 1. 把 `detectQuestionStrategy()` 收回 v7

不再从 `rag5/context-builder.mjs` 里引入。

这样版本 7 的策略判断现在就地可读：

- job_match
- skill
- project
- experience
- general

学习时不用跨文件回跳。

#### 2. 把 `denoiseMatches()` 收回 v7

不再继续依赖 `rag4/rag-pipeline-v4.mjs`。

现在 v7 自己维护：

- 噪音原因识别
- 去噪保留规则
- 最小结果兜底

这样版本 7 在“策略识别 → 重排 → 去噪 → 分层”这条链路上，  
已经基本形成了自己的闭环。

## `selectFinalMatchesV7()` 为什么继续拆小

这次另一个反馈是：

> `selectFinalMatches` 读配置、推 preferred sections、去噪、证据分层、数量限制全塞在一起，太长了。

这个反馈是成立的。

因为这种函数虽然“逻辑完整”，  
但它会让阅读者在一个函数里不断切换脑内视角：

- 现在是在读配置
- 还是在做分类
- 还是在做数量控制

### 这次怎么拆

版本 7 这次没有把它拆成很多零碎模块，  
而是只拆出几块真正能提升理解效率的小函数：

- `resolvePreferredSections()`：专门负责从 boost 配置推导优先 section
- `buildSelectionParams()`：专门负责把多份配置收口成一组分层参数
- `classifyEvidence()`：专门判断一条记录属于 primary / support / cross_support 哪一类
- `selectTierMatches()`：专门负责“按层级筛选 + 打 tier 标签”

这样做的好处是：

- 没有过度拆文件
- 但 `selectFinalMatchesV7()` 主体明显更短
- 阅读时更容易看到它的大结构

## `hardDropMinNoiseReasons` 的语义如何理解

这个命名本身确实不够直观。

原来的判断写法是：

```js
noiseReasons.length < hardDropMinNoiseReasons
```

如果配置里写的是：

```json
"hardDropMinNoiseReasons": 4
```

很多人第一次看都会问：

> 这是“少于 4 个可以留”，还是“等于 4 个才丢”？  
> 这个数值到底怎么解释？

### 这次没有直接改配置名，但做了两层补救

#### 1. 在代码里补了语义注释

现在在 `buildSelectionParams()` 附近，已经明确说明：

- `4` 的意思不是“必须有 4 个噪音原因才能入选”
- 而是：
  - **噪音原因达到 4 个及以上，直接视为硬丢弃**
  - 也就是说：
    - `0 / 1 / 2 / 3` 个噪音原因，仍然允许继续参与主证据 / 辅助证据判断

#### 2. 在文档里补这层业务解释

这样即使 JSON 里本身不能写注释，  
学习者也能在文档和代码注释里读懂这个阈值的含义。

### 为什么这次不直接改配置字段名

因为当前这一轮的重点是：

- 先去掉跨版本耦合
- 先把主流程读起来更顺

如果这时又顺手把：

- `hardDropMinNoiseReasons`

改成一套全新的配置命名，  
会让这轮改动的范围继续膨胀。

所以这次采取的是更保守的处理：

> **先补清语义，再等后续如果确实要大改配置结构时，再统一重命名。**

## 继续收口 v7 的 ask 脚本：参数解析、空层级日志与流式提示

这轮继续看的不是 `context-builder`，  
而是版本 7 的 CLI 入口：

- `ask-resume-7-progressive-ask.mjs`

这一层虽然不直接决定检索质量，  
但它会明显影响：

- 初次运行时的理解成本
- 调试日志的可读性
- CLI 使用体验

### 1. `parseArgs` 继续手写已经不划算了

这一点和前面几个版本一样：

- 参数变多后
- 自己维护 `for` 循环
- 每加一个参数都要继续堆 `if`

会让脚本入口越来越臃肿。

#### 这次怎么改

这次直接切到 Node 内置的：

- `node:util -> parseArgs`

好处是：

- 不需要额外引第三方库
- 解析规则更集中
- 位置参数和长选项都更清晰

对于当前这个学习 demo 来说，这已经比继续手写更合适。

---

### 2. `onAnswerToken` 加上防御式处理

虽然正常情况下流式 token 应该是字符串，  
但 CLI 层最好还是做一层更稳妥的保护：

- 没内容就直接跳过
- 真正写入时统一 `String(text)`

这样即使上游后面换模型、换 SDK、或者 chunk 结构稍有变化，  
CLI 这里也不至于因为 `stdout.write(undefined)` 这类问题直接报错。

---

### 3. 空的 `cross_support` / `reserve` 层级不再打印

版本 7 引入了更多证据层级，这是好事。  
但如果某一层当前为空，还继续打印：

- 一个标题
- 一个空块

日志会显得比较松散。

#### 这次怎么改

增加了一个很小的包装函数：

- `printMatchesIfAny()`

规则是：

- 有内容才打印
- 空数组直接跳过

这样调试日志就更聚焦于“当前这轮真正出现了什么层级”。

---

### 4. 给流式模式补一条用户提示

版本 7 默认是：

- `streamAnswer = true`

这本身是为了提升“等待感”，  
但第一次使用时也容易让人有一点心理落差：

- 为什么回答先出来了？
- Raw candidates / Final matches 怎么在后面？

#### 这次怎么改

在 CLI 启动时补一条明确提示：

> 调试信息将在回答完成后打印，先展示实时回答内容

这样用户会更容易理解当前输出顺序。

---

## 这一轮的核心结论

这次的修改都不大，  
但它们很适合学习阶段：

- `parseArgs` 从手写切到标准库
- `onAnswerToken` 更稳
- 空层级不再打扰日志
- 流式模式有了更明确的引导

一句话总结就是：

> **版本 7 这轮不是继续“加能力”，而是在把 CLI 入口层打磨得更稳、更清晰、更适合初学者阅读和运行。**

---

## 继续收口 v7：补 TSDoc 注释，并做成真正独立的版本

到了这一轮，版本 7 虽然主链路已经基本形成闭环，  
但从“学习仓可读性”的角度看，还剩两个明显问题：

1. **核心代码解释还不够系统**
2. **版本目录还没有真正完全闭合**

### 1. 为什么这轮要补 TSDoc / 核心注释

前面几轮我们已经在代码里陆续补了很多业务注释，  
但它们更偏：

- 针对某一段判断的解释
- 针对某一轮修改的背景说明

这还不够。

因为当学习者第一次进入版本 7 时，  
他真正最需要的是两层信息：

- **这个函数整体负责什么**
- **这个函数的输入、输出、失败条件是什么**

所以这次补注释的原则不是“逐行讲废话”，  
而是：

- 对导出函数统一补 TSDoc / JSDoc 风格块注释
- 对关键内部函数补高价值的设计说明
- 把文档里特别强调过的注意点，直接落到对应代码旁边

例如这次特别补回代码的点包括：

- `hardDropMinNoiseReasons` 的真实语义
- `getMatchKey()` 为什么不能只靠 `id`
- `rag-pipeline-v7` 只是编排器，而不是检索层
- `ask` 层负责 CLI 表现，表现层逻辑不要回塞到 pipeline
- `cross_support` 是补证，不是抢主位
- `reserve` 是兜底补充，不是默认主证据

这样做的意义是：

> **学习者既能从文档看整体演进，也能在代码里就地读到关键设计意图。**

---

### 2. 为什么这次不再满足于“只去掉函数级跨版本依赖”

前面一轮我们已经把：

- 策略识别
- 去噪逻辑

收回到了版本 7 内部。

但继续往下看会发现，  
版本 7 仍然还在依赖：

- 旧版本的历史存储模块
- 旧版本的 keyword hints 配置
- 旧版本的 section boost 配置
- 旧版本的 rerank 配置
- 旧版本的 precision 配置
- 非版本目录里的通用 retriever

这说明它虽然“主逻辑”已经比较独立，  
但还不算真正意义上的：

> **版本内闭合。**

如果继续保留这种状态，会有几个问题：

- 读 v7 时还得频繁跳去别的目录
- 后面改旧版本配置时，v7 可能被被动影响
- 学习者会误以为“v7 已经独立”，但实际上关键参数和模块还散在别处

所以这次的处理不是继续抽 shared common，  
而是更直接地：

- **把版本 7 运行真正依赖的模块和配置，整体复制收回到 `src/rag7/` 内部**

这次选择“复制收回”而不是“抽 shared common”的原因也很明确：

- 当前仓库是学习型渐进式 demo
- 每一代版本本身就是教学载体
- 对这种仓库来说，“每一代可单独阅读”比“共用一份抽象层”更重要

换句话说，这次优先的是：

> **版本可读性与版本独立性，而不是最大化代码去重。**

---

### 3. 这次具体收回了什么

这次版本 7 在目录内补齐了：

- `history-store-v7.mjs`
- `retriever-v7.mjs`
- `config/keyword-hints.json`
- `config/section-boost-config.json`
- `config/rerank-config.json`
- `config/precision-config.json`

这样现在 `ask:v7` 与 `rag7/*` 主体运行时，  
不再需要继续回头引用旧版本目录。

这一步之后，版本 7 真正做到了：

- CLI 入口在 v7
- RAG pipeline 在 v7
- context-builder 在 v7
- prompt-builder 在 v7
- prompt 模板在 v7
- selection 配置在 v7
- rerank / precision / section boost / keyword hints 配置也都在 v7
- session history 存储也在 v7

从学习体验上说，这个变化其实非常重要：

> **现在你可以把 `src/rag7/` 看成一个可以单独阅读、单独维护的完整实现。**

### 4. 顺手修掉的历史目录路径债

这次在收回 `history-store` 时，  
还顺手处理了一个旧遗留问题：

- 默认历史目录原来仍然指向旧 demo 时代的 `examples/.../.runtime/chat-sessions`

这个路径在项目早期没问题，

因为它会继续暴露一个过时事实：

> **“这个项目仍然像是某个旧 demo 的附属目录。”**

所以这次直接把默认目录切成当前仓库根目录下的：

- `.runtime/chat-sessions`

这样更符合当前仓库的真实身份，也更符合版本 7 独立化之后的结构。

## 这一轮补注释 + 独立化之后的判断

这一轮并没有改业务阈值，  
也没有继续扩充选择策略。

它做的事情更像是：

- 让版本 7 的运行依赖完全收口
- 让核心代码读起来更有“完整版本”的感觉
- 让文档与代码里的设计意图更一致

一句话总结就是：

> **从这一轮开始，v7 不再只是“当前最新实现”，而是一个真正能单独拿出来学习、讲解和维护的完整版本。**
